drift check against `848e31d` and reconcile all prerequisite changes. Never

## Roadmap revision

On 2026-07-16, the operator chose not to establish versioned migrations while the application remains in its
internal flow and production schema changes continue through `drizzle-kit push`. Plan 008 alone is skipped.
Plans that originally depended on it remain necessary and have been remodelled to validate schema changes by
pushing the live schema into guarded disposable PostgreSQL databases. A production push remains an explicit
operator action with a backup, change record, and write pause when old and new writers are incompatible.

Plans 001-010 and 014 are complete except for intentionally skipped Plan 008. Continue with the remaining plans
in index order: 011, 012, 013, 015, and 016.

## Recommended execution waves

These waves respect dependencies; plans within a wave may use separate
worktrees, but should not edit the same worktree concurrently.

1. **Immediate boundaries and baseline:** 001, 002, 003, 005, and 006.
2. **Auth invariant and test foundation:** 004, 007, and 014.
3. **Independent inventory safety:** 010. Plan 008 is intentionally skipped.
4. **Order trust boundary and dependency cohorts:** 009 and 012.
5. **Retry safety and aligned request limits:** 011 and 015.
6. **Mutation performance and reproducible setup:** 013 and 016.

Plans 009 and 011 extend the live Drizzle schema and prove each checkpoint by
using Plan 007's guarded fresh-database push harness. Plan 012 preserves the
hardened auth policies from Plans 001/004 and verifies the same PostgreSQL suite
after upgrading Better Auth and Drizzle. Plan 016 runs after that dependency
cohort so its first-admin bootstrap is proven against the final installed auth
API.

## Shared guardrails

- Use `pnpm`; keep changes within each plan's explicit scope.
- Treat all database URLs and credentials as secrets. Database mutation tests
  use only newly created, exact-name, loopback disposable databases.
- Schema-changing plans use `drizzle-kit push` only against guarded disposable
  databases during implementation. A shared or production push requires an
  explicit operator-approved backup, change record, compatibility review, and
  rollout window; plans never mutate production as a verification step.
- Do not push, deploy, run Trigger tasks, or mutate a shared environment unless
  a plan explicitly requires an approved operator action.
- Preserve user changes and update only the executing plan's status row here.
