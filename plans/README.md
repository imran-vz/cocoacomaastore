## Roadmap revision

On 2026-07-16, the operator chose not to establish versioned migrations while the application remains in its
internal flow and production schema changes continue through `drizzle-kit push`. Plan 008 and every plan that
transitively depends on it are therefore skipped. Their plan files remain available for reconsideration; their
dependency metadata is unchanged.

The remaining implementation order is Plan 010 followed by Plan 014.

## Recommended execution waves

These waves respect dependencies; plans within a wave may use separate
worktrees, but should not edit the same worktree concurrently.

1. **Immediate boundaries and baseline:** 001, 002, 003, 005, and 006.
2. **Auth invariant and test foundation:** 004, 007, and 014.
3. **Database foundation and independent inventory safety:** 008 and 010.
4. **Order trust boundary and dependency cohorts:** 009 and 012.
5. **Retry safety and aligned request limits:** 011 and 015.
6. **Mutation performance and reproducible setup:** 013 and 016.

Plan 008 deliberately converts Plan 007's temporary fresh-schema push into
tracked migration replay. Plans 009 and 011 then extend that history with
forward migrations. Plan 012 preserves the hardened auth policies from Plans
001/004 and verifies the established migration-backed PostgreSQL suite after
upgrading Better Auth and Drizzle. Plan 016 runs after that dependency cohort so
its first-admin bootstrap is proven against the final installed auth API.

## Shared guardrails

- Use `pnpm`; keep changes within each plan's explicit scope.
- Treat all database URLs and credentials as secrets. Database mutation tests
  use only newly created, exact-name, loopback disposable databases.
- Tracked migration files are immutable after application. Create forward
  migrations; never improvise a production migration-ledger write.
- Do not push, deploy, run Trigger tasks, or mutate a shared environment unless
  a plan explicitly requires an approved operator action.
- Preserve user changes and update only the executing plan's status row here.

## Findings considered but not given separate plans

- The chart's `dangerouslySetInnerHTML` input is code-owned static CSS; no
  untrusted-data path was found, so it was not reported as XSS.
- Current-day live reporting versus closed-day compiled analytics is a valid
  boundary. The stale inline-recompute documentation is corrected inside Plan
  013 instead of becoming a second analytics plan.
- Duplicate dessert-fetch clients, broad list queries, and unused direct
  dependencies were lower-leverage observations without enough production
  volume/reachability evidence for separate findings.
- Stock-demand insights, modifier-level revenue attribution, and a missing-data
  analytics action remain product/architecture directions. They require product
  decisions and are intentionally outside these 16 implementation plans.
